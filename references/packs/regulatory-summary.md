# Regulatory Summary Reference Pack

Detailed document-summary framework for
`skills/specialists/regulatory-summary.md`. Load this pack when preparing,
tailoring, or quality-checking a regulatory document summary.

This pack describes extraction and drafting discipline. It is not a cached
regulatory source.

## Quick Start

1. Identify the document type and procedural posture.
2. Extract metadata before summarizing requirements.
3. Pull provisions with pinpoint citations.
4. Separate binding text from guidance, background, and agency reasoning.
5. Track effective dates, comment deadlines, and compliance deadlines.
6. Convert requirements into action items only when the document supports them.

## Document-Type Matrix

| Document type | Binding posture | Required special handling |
|---|---|---|
| Final rule | Usually binding after effective date | Extract effective date, compliance deadline, amended CFR/code sections, and transition rules. |
| Proposed rule / NPRM | Not binding | Capture comment deadline, questions asked, proposed text, and likely affected entities without current-law phrasing. |
| Interim final rule | Binding but procedurally open | Capture effective date, comment deadline, and possible later changes. |
| Guidance / FAQ | Interpretive unless binding authority exists | Identify legal effect and whether it supersedes prior guidance. |
| Enforcement order | Binding on parties; broader signal varies | Capture conduct, legal basis, remedy, admissions/denials, and finality. |
| No-action letter | Limited reliance | Capture addressee, facts assumed, conditions, revocation limits, and non-precedential language. |
| Advisory opinion | Depends on authority | Capture facts, question presented, legal conclusion, and reliance limits. |
| Compliance framework | Often best-practice or safe-harbor adjacent | Identify whether law incorporates it or merely recommends it. |
| Administrative decision | Binding effect varies | Capture tribunal, posture, holding, appeal status, and remedy. |

|---|---|---|---|---|---|---|
| 1 | | | | | | |

Rules:

- Every requirement or prohibition needs a pinpoint citation.
- Use direct quotations for definitions, legal standards, deadlines, and safe
  harbors when exact wording matters.
- Do not turn background discussion into an obligation.
- Mark ambiguity with `[AMBIGUOUS]` and state what further source would resolve
  it.

## Output Quality Checklist

- [ ] Document type and procedural posture are explicit.
- [ ] Metadata table is complete or uses `Not stated`.
- [ ] Every substantive requirement has a pinpoint citation.
- [ ] Binding obligations are separated from guidance and background.
- [ ] Deadlines are separated by type.
- [ ] Prior-framework deltas are included when relevant.
- [ ] Safe harbors list conditions and limits.
- [ ] Ambiguities are marked instead of resolved by inference.
- [ ] Compliance actions trace to source-backed provisions.

## Pitfalls

- Treating a proposed rule as binding.
- Treating guidance as binding without legal-effect analysis.
- Omitting transition periods.
- Converting agency policy discussion into mandatory action items.
- Ignoring reliance limits in no-action letters or advisory opinions.
- Summarizing enforcement penalties without the conduct and legal basis.
- Filling missing metadata from commentary rather than the document itself.
